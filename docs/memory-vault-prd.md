# MemoryVault — 产品设计文档（PRD）

> **AI 时代的 1Password：你只需要调教一次，所有 AI 都认识你。**

---

## 一、产品概述

### 1.1 产品定位

MemoryVault 是一个**独立于任何大模型和 SaaS 平台之外的"个人 AI 上下文基础设施"**。它在用户与 AI 的日常交互中自动提炼、加密存储并跨平台同步用户的工作记忆，使用户在任何 AI 工具中都能获得个性化、连续性的体验。

### 1.2 一句话价值主张

**你的 AI 记忆属于你，不属于任何平台。换模型、换工具，记忆跟着你走。**

### 1.3 产品愿景

成为 AI Agent 生态的"记忆中间件"——就像 1Password 把密码从浏览器中解放出来一样，MemoryVault 把用户的 AI 上下文从大模型公司中解放出来。

## 二、目标用户

### 2.1 核心用户画像

| 画像 | 描述 | 核心痛点 |

- 企业级团队协作（v2 再考虑）

### 3.1 用户痛点

| # | 痛点 | 严重程度 | 现有替代方案 |
|---|------|----------|-------------|
| P1 | **反复调教**：换一个 AI 工具就要重新解释自己是谁、在做什么、偏好什么 | 高频 | 手动在每个工具里维护 system prompt / CLAUDE.md |

| 跨平台同步 | 记忆加密存储在云端，在任意设备和 AI 工具中可用 | P0 |

## 五、核心功能设计

### 5.1 记忆的生命周期

```
观察 → 提取 → 分类 → 存储 → 整理 → 检索 → 注入 → 遗忘
```

### 5.2 记忆分类体系

| 记忆类型 | 说明 | 示例 | 存储方式 |
|---------|------|------|---------|
| **身份记忆 (Identity)** | 用户是谁、做什么工作 | "全栈开发者，主要用 TypeScript" | 知识图谱 |
| **偏好记忆 (Preference)** | 用户的工具和风格偏好 | "代码风格偏好函数式，不喜欢 class" | 知识图谱 |
| **项目记忆 (Project)** | 特定项目的背景和架构 | "项目 A 用 Next.js + Supabase，monorepo" | 知识图谱 + 向量 |
| **情景记忆 (Episode)** | 具体的问题解决过程 | "上周 Docker 网络问题的排查过程" | 向量数据库 |
| **规则记忆 (Rule)** | 用户明确要求 AI 遵守的规则 | "永远不要用 var，只用 const/let" | 知识图谱 |

### 5.3 自动记忆提炼（核心功能）

#### 5.3.1 提炼时机

| 触发条件 | 说明 |
|---------|------|
| 对话结束时 | Agent 会话关闭后，后台异步分析本次对话，提取有价值的记忆碎片 |
| 关键行为时 | 用户连续纠正 AI 的某个行为（如反复改代码风格），实时捕获为偏好 |
| 定时整理 | 每天凌晨跑一次"记忆整理员"，合并去重，短期记忆→长期记忆 |

#### 5.3.2 提炼 Prompt 设计（核心）

```yaml
system: |
  你是一个记忆提炼引擎。分析以下用户与 AI 的对话记录，
  提取值得长期记住的信息。

  提取规则：
  1. 只提取"跨会话有价值"的信息，忽略一次性的具体问题
  2. 关注用户的偏好、习惯、纠正行为和反复出现的模式
  3. 关注项目层面的架构决策和技术选型
  4. 忽略通用知识（如"React 是一个前端框架"）
  5. 如果与已有记忆冲突，标记为"更新"而非新增

  输出格式：
  - type: identity | preference | project | episode | rule
  - content: 记忆内容（一句话）
  - confidence: high | medium | low
  - source: 来源对话摘要
  - tags: [相关标签]
```

#### 5.3.3 记忆冲突处理

当新提取的记忆与已有记忆冲突时：

| 场景 | 处理策略 |
|------|---------|
| 新偏好覆盖旧偏好 | 保留新的，旧的标记为"历史版本"（可回溯） |
| confidence: low 的新记忆 vs 已有记忆 | 不覆盖，标记为"待确认"，累计出现 3 次后自动升级 |
| 用户明确说"我现在改用 X 了" | 立即更新，高置信度 |

### 5.4 记忆检索与注入

#### 5.4.1 检索策略

当 Agent 发起一次新对话时，Memory Server 自动执行：

```
Step 1: 提取当前对话的关键词和意图
Step 2: 从知识图谱中拉取相关的身份 / 偏好 / 规则（精确匹配）
Step 3: 从向量数据库中检索相关的情景记忆（语义匹配，Top-K）
Step 4: 按相关性排序，拼装为 context block
Step 5: 注入到 Agent 的 system prompt 或 context window 中
```

#### 5.4.2 注入格式（MCP Resource）

```markdown
## User Memory Context (auto-injected by MemoryVault)

### Identity
- 全栈开发者，5 年经验，主力语言 TypeScript
- 当前在独立开发一个 SaaS 产品

### Preferences
- 代码风格：函数式优先，避免 class
- 框架：Next.js + TailwindCSS
- 测试：偏好 Vitest，不用 Jest

### Current Project: ProjectX
- 技术栈：Next.js 14 + Supabase + Stripe
- 架构：monorepo (Turborepo)
- 当前进度：正在做用户鉴权模块

### Relevant Episodes
- [3 天前] 解决了 Supabase RLS 策略导致的权限 Bug
- [1 周前] 决定从 Prisma 迁移到 Drizzle ORM
```

### 5.5 记忆管理 Dashboard

提供一个轻量的 Web UI，让用户掌控自己的记忆：

#### 5.5.1 核心页面

| 页面 | 功能 |
|------|------|
| **记忆时间线** | 按时间倒序展示所有记忆条目，支持按类型 / 项目 / 标签筛选 |
| **记忆图谱** | 可视化展示知识图谱中的实体关系（交互式节点图） |
| **记忆编辑** | 查看、修改、删除任意记忆条目；手动添加新记忆 |
| **数据导出** | 一键导出全部记忆为 Markdown 文件夹 / JSON / CSV |
| **接入管理** | 管理哪些 AI 工具有权读取 / 写入你的记忆，支持撤销授权 |
| **记忆健康度** | 展示记忆总量、分类分布、最近更新情况、冲突待确认项 |

#### 5.5.2 交互设计要点

- **审批机制**：首次使用时，每条新提炼的记忆弹出确认（类似 1Password 的"是否保存此密码"）。用户信任系统后，可切换为"自动保存 + 定期审阅"模式。
- **记忆可解释性**：每条记忆都附带"为什么记住这个"的来源说明和原始对话片段链接。
- **快速纠错**：用户看到一条错误记忆，一键删除 + 反馈"这个不对"，系统学习避免类似错误。

## 六、MCP 协议接口设计

### 6.1 Tools（Agent 可调用的能力）

```typescript
// 写入记忆
memory_write({
  content: string,        // 记忆内容
  type: MemoryType,       // identity | preference | project | episode | rule
  tags?: string[],        // 标签
  project?: string,       // 关联项目
  confidence?: number,    // 置信度 0-1
  source?: string,        // 来源描述
})

// 搜索记忆
memory_search({
  query: string,          // 自然语言查询
  type?: MemoryType,      // 可选：限定类型
  project?: string,       // 可选：限定项目
  limit?: number,         // 返回数量，默认 10
})

// 删除 / 遗忘
memory_forget({
  id: string,             // 记忆 ID
  reason?: string,        // 遗忘原因（用于学习）
})

// 批量更新（记忆整理员使用）
memory_consolidate({
  merge: string[],        // 要合并的记忆 ID 列表
  into: string,           // 合并后的内容
})
```

### 6.2 Resources（自动注入的上下文）

```typescript
// Agent 启动时自动加载
memoryvault://context/current
// → 返回与当前对话最相关的记忆上下文（Markdown 格式）

// 特定项目的记忆
memoryvault://project/{project_name}
// → 返回该项目的架构、技术栈、历史决策等
```

### 6.3 Prompts（预置的提示模板）

```typescript
// 记忆提炼：对话结束后调用
memoryvault://prompts/extract
// → 分析本次对话，提取值得记忆的信息

// 记忆审阅：让用户审阅最近的记忆
memoryvault://prompts/review
// → 展示最近 7 天的新记忆，请用户确认 / 修改 / 删除
```

## 七、数据模型

### 7.1 记忆条目（Memory Entry）

```typescript
interface MemoryEntry {
  id: string                    // UUID
  type: 'identity' | 'preference' | 'project' | 'episode' | 'rule'
  content: string               // 记忆内容（自然语言）
  tags: string[]                // 标签
  project?: string              // 关联项目名
  confidence: number            // 置信度 0-1
  source: {
    tool: string                // 来源工具（如 "cursor", "claude-desktop"）
    conversation_id?: string    // 原始对话 ID
    excerpt?: string            // 原始对话片段
  }
  embedding: number[]           // 向量嵌入（用于语义检索）
  created_at: string            // ISO 8601
  updated_at: string
  expires_at?: string           // 可选过期时间（情景记忆可设置自动遗忘）
  status: 'active' | 'archived' | 'pending_review'
  versions: MemoryVersion[]     // 历史版本（记忆被更新时保留旧版本）
}

interface MemoryVersion {
  content: string
  updated_at: string
  reason: string                // 更新原因
}
```

### 7.2 知识图谱节点

```typescript
interface GraphNode {
  id: string
  label: string                 // 如 "TypeScript", "ProjectX", "User"
  type: 'person' | 'technology' | 'project' | 'concept' | 'preference'
  properties: Record<string, any>
}

interface GraphEdge {
  from: string                  // Node ID
  to: string                   // Node ID
  relation: string             // 如 "prefers", "uses", "works_on", "decided"
  weight: number               // 关系强度
  source_memory_id: string     // 关联的记忆条目 ID
}
```

## 八、安全设计

### 8.1 端到端加密方案

```
用户设备                          云端
┌──────────┐                  ┌──────────┐
│ 生成主密钥 │                  │          │
│ (Master   │                  │ 只存储    │
│  Key)     │                  │ 密文数据  │
│           │                  │          │
│ 记忆明文   │──── 加密 ────▶  │ 加密记忆  │
│           │                  │          │
│ 向量嵌入   │──── 加密 ────▶  │ 加密向量  │
│ (本地生成) │                  │ (盲检索)  │
│           │                  │          │
│ 解密 ◀────│──── 同步 ────── │          │
└──────────┘                  └──────────┘
```

### 8.2 安全原则

| 原则 | 说明 |
|------|------|
| 零知识 | 服务端永远看不到用户记忆明文 |
| 本地加密 | 所有加密 / 解密操作在用户设备上完成 |
| 密钥不上云 | 主密钥仅存储在用户本地设备（或用户自己的密码管理器） |
| 可审计 | 每次记忆读写操作都记录在本地审计日志中 |
| 选择性共享 | 用户可以选择哪些记忆类型允许云端同步，哪些仅本地 |

### 8.3 MVP 阶段简化方案

完整的密文向量检索是学术级难题。MVP 阶段的务实方案：

- 文本记忆：标准 AES-256 加密存储于云端，检索时下载到本地解密后执行
- 向量嵌入：本地生成、本地存储，仅通过增量同步的方式跨设备同步加密后的向量文件
- 长期方案：关注 Homomorphic Encryption 和 Encrypted Vector Search 的学术进展

## 九、技术选型建议

### 9.1 MVP 技术栈

| 组件 | 推荐方案 | 备选 | 理由 |
|------|---------|------|------|
| MCP Server | TypeScript + @modelcontextprotocol/sdk | Python | 生态最成熟，示例最多 |
| 向量数据库 | Supabase pgvector（云端） 或 SQLite + sqlite-vss（本地优先） | Qdrant / Pinecone | Supabase 免费额度够用；本地优先方案更符合安全理念 |
| 知识图谱 | 轻量方案：JSON-LD 文件 + 内存图查询 | Neo4j Aura | MVP 阶段数据量小，不需要重型图数据库 |
| 向量嵌入 | OpenAI text-embedding-3-small 或 本地 all-MiniLM-L6-v2 | Cohere Embed | 本地嵌入模型更符合隐私理念 |
| 记忆提炼 LLM | Claude 3.5 Haiku / GPT-4o-mini | 本地小模型 | 成本低，速度快，质量够用 |
| 前端 Dashboard | Next.js + TailwindCSS | — | 快速开发 |
| 云端同步 | Supabase Realtime / CRDTs (Yjs) | Firebase | 支持离线优先 + 冲突解决 |
| 加密 | Web Crypto API + libsodium | — | 标准化，跨平台 |

### 9.2 部署架构

```
本地优先 (Local-first) 架构：

用户设备
├── MemoryVault 本地引擎（常驻后台进程）
│   ├── SQLite 本地数据库（记忆明文 + 向量）
│   ├── MCP Server（监听本地端口）
│   └── 加密 / 解密模块
│
└── 云端同步（可选）
    ├── Supabase（加密数据同步）
    └── 用户认证（Magic Link / Passkey）
```

## 十、商业模式

### 10.1 定价策略

| 层级 | 价格 | 包含内容 |
|------|------|---------|
| **Free** | $0 | 本地存储，不限容量；单设备；MCP 接入 2 个工具 |

- 本地功能永久免费 → 积累用户基数

- 团队版为长期增长引擎 → 企业愿意为"团队 AI 上下文共享"付费

- [ ] CLI 工具：手动添加 / 查看 / 删除记忆
- [ ] Markdown 导出功能

**验证指标：** 用户在工具 A 产生的记忆，能否在工具 B 中被有效利用？

### Phase 2：体验打磨（第 5-12 周）

**目标：让记忆提炼"真的好用"**

- [ ] 记忆提炼质量优化（什么该记、什么不记）

## 十三、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| MCP 协议未被广泛采纳 | 跨工具接入受限 | 同时支持 REST API + 浏览器插件作为后备接入方式 |
| 记忆提炼质量差 | 用户失去信任 | 初期采用"人工审批"模式，逐步过渡到自动化；建立反馈循环 |
| 加密向量检索性能差 | 云端体验卡顿 | MVP 采用"本地检索 + 云端同步"的折中方案 |
| 大模型公司自建记忆层 | 市场被挤压 | 差异化定位：强调"模型无关"和"数据主权"；开源社区建设 |
| 用户隐私顾虑 | 不敢使用 | 开源核心代码；提供 self-hosted 选项；零知识加密 |

## 十四、竞品分析

| 产品 | 定位 | 优势 | 劣势 | 与 MemoryVault 的差异 |
|------|------|------|------|----------------------|
| **Mem0** | AI Agent 记忆层 (To B) | 开源；支持多种 LLM | 面向企业，个人开发者体验弱；无 E2EE | MemoryVault 面向 To C，强调数据主权和加密 |
| **Zep** | LLM 长期记忆引擎 | 自动摘要；知识图谱 | 深度绑定 LangChain 生态；无跨工具能力 | MemoryVault 基于 MCP，不绑定任何框架 |
| **Claude Memory** | Claude 内置记忆 | 原生集成，无需配置 | 仅限 Claude 生态；不可导出 | MemoryVault 跨模型、可导出、用户拥有 |
| **ChatGPT Memory** | GPT 内置记忆 | 自动提取偏好 | 仅限 OpenAI 生态；不透明 | 同上 |
| **Obsidian + AI** | 个人知识库 | 本地存储；插件生态强 | 需手动维护；不自动提炼 | MemoryVault 自动化程度更高 |
